The Problem (Recap)

OneTake's Next.js frontend runs on Vercel. Vercel uses dynamic IPs that can't be whitelisted. This means the frontend cannot query Azure PG or the OneForma prod DB directly. We need a secure path for dashboard data without exposing any database to the public internet.

Proposed Solution: Internal API Proxy (Zero Public DB Exposure)

Instead of opening database access from the outside, we've built a lightweight API proxy that runs inside the existing Azure Container App (onetake-py-worker). The proxy sits on the internal Azure network, queries the databases locally, and returns JSON results over HTTPS with Bearer token authentication.

🔒 Key security point: No database is exposed to the public internet. The only public-facing component is an HTTPS API endpoint on the Container App, protected by Bearer token auth. All database traffic stays internal to the Azure network.

Vercel ──✗──→ Azure PG       (IP whitelist blocks)
Vercel ──✗──→ Prod DB        (private network, no public access)

Security Controls

LayerControlDetail
Authentication Bearer token on every request 256-bit secret shared only between Vercel (env var) and Container App (env var). Rejected with 401 if missing/invalid.
Transport HTTPS only Container Apps provides TLS termination automatically. No plaintext traffic.
Database access No public exposure Both Azure PG and prod DB remain on private networks. Only the proxy process (inside the Container App) connects to them.
Read-only scope Proxy returns JSON, not raw SQL The proxy exposes specific endpoints (list projects, get funnel data, get channel links). No arbitrary query execution. No write operations except a materialized view refresh.
Prod DB access Read-only connection string We'd use the .reader endpoint (oneformapostgresprod.reader.postgres.database.azure.com) — inherently read-only.
Network Internal only DB connections stay within Azure VNet / private network. No database port is exposed externally.

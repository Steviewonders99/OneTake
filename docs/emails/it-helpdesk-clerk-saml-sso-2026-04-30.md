## What We Need

A **SAML Enterprise Application** in Azure AD (Entra ID) so the OneForma marketing team can sign into OneTake using their existing Microsoft credentials (@centific.com / @oneforma.com accounts).

Our application uses **Clerk** (clerk.com) as our authentication provider. Clerk acts as the SAML Service Provider (SP) and needs to be connected to Centific's Azure AD as the Identity Provider (IdP).

## Setup Steps for Azure AD Admin

### Step 1: Create Enterprise Application

1. Go to **Azure Portal** → **Microsoft Entra ID** → **Enterprise Applications**
2. Click **New Application** → **Create your own application**
3. Name: `OneTake - Clerk SSO`
4. Select: **Integrate any other application you don't find in the gallery (Non-gallery)**
5. Click **Create**

### Step 2: Assign Users / Groups

Under the Enterprise Application → **Users and groups** → **Add user/group**:

Assign the following users or groups who should have access:

- Steven Junop (steven.junop@centific.com)
- Jennifer / Jen (marketing team — Steven can provide full list)
- Any group containing OneForma marketing + recruitment team members

Alternatively, if a security group exists for the OneForma marketing team, assign that group.

### Step 3: Configure SAML

Go to **Single sign-on** → Select **SAML** → **Edit Basic SAML Configuration**:

| Field | Value |
|---|---|
| **Identifier (Entity ID)** | *[Steven will provide — generated by Clerk]* |
| **Reply URL (ACS URL)** | *[Steven will provide — generated by Clerk]* |
| **Sign on URL** | `https://onetake.oneforma.com/sign-in` |
| **Relay State** | Leave blank |
| **Logout URL** | Leave blank |

> **Note:** I will send the exact Entity ID and Reply URL values in a follow-up email once they are generated in our Clerk dashboard. The Enterprise Application can be created and users assigned before these values are configured.

### Step 4: Verify Attribute Mappings

Under **Attributes & Claims**, ensure these three claims are mapped:

| Claim | Name | Source Attribute |
|---|---|---|
| **Email** (required) | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress` | `user.mail` |
| **First name** | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname` | `user.givenname` |
| **Last name** | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname` | `user.surname` |

These are typically pre-configured by default in Azure AD SAML setups.

### Step 5: Send Back to Steven

Once the Enterprise Application is created, please send me:

1. **App Federation Metadata URL** — found under **SAML Certificates** → **App Federation Metadata Url** (looks like: `https://login.microsoftonline.com/<tenant-id>/federationmetadata/2007-06/federationmetadata.xml?appid=<app-id>`)
2. Confirmation that the application is created and users are assigned

I will paste the Metadata URL into Clerk to complete the connection on our side.

Request 1: Enable Container App Ingress (Port 8080)

P0 — Blocks Thursday demo

What: Enable external HTTP ingress on the onetake-py-worker Container App, port 8080.

Why: We've built a lightweight JSON API proxy (db_proxy.py) that runs inside the Container App alongside the existing worker. Our Next.js frontend is deployed on Vercel, which has dynamic IPs that can't be whitelisted on the Azure PG firewall. The proxy solves this — it sits inside the Container App (same network as PG), accepts HTTPS requests from Vercel, queries PG, and returns JSON. Bearer token authentication is enforced on every request.

Architecture:

# Current (broken)
Vercel (dynamic IPs) ──✗──→ Azure PG (IP whitelist blocks)

# With proxy (fixed)
Vercel ──HTTPS──→ Container App :8080 ──direct──→ Azure PG
                  (db_proxy.py)         (same network, no IP issue)
                  Bearer token auth

What I need:

SettingValueNotes
Container App onetake-py-worker Existing app in onetake-rg
Ingress Enabled, External Must be reachable from public internet (Vercel)
Target port 8080 The proxy listens on 8080
Transport HTTP (TLS terminated by Container Apps) Container Apps provides the SSL certificate automatically

CLI equivalent (if you prefer):

az containerapp ingress enable \
  --name onetake-py-worker \
  --resource-group onetake-rg \
  --target-port 8080 \
  --type external \
  --transport http
Security note: The proxy enforces Bearer token authentication on every request via the PROXY_SECRET environment variable. No unauthenticated access is possible. The token is shared only between Vercel (as an env var) and the Container App.

After ingress is enabled, I'll need the Container App's FQDN (e.g., onetake-py-worker.<env>.<region>.azurecontainerapps.io) to configure Vercel.

Request 2: Allowlist pg_trgm Extension on Azure PG

P1 — Blocks Azure PG migration

What: Add pg_trgm to the allowed extensions list on onetake-pg-west01.

Why: We built a Project Registry system that uses fuzzy text matching to automatically link WordPress job posts to their advertising campaigns, email sequences, recruiter links, and organic social posts. The pg_trgm extension enables this fuzzy matching — it's a built-in PostgreSQL extension (not third-party) that ships with every PG installation.

What it does: Trigram-based text similarity. For example, it can match "hummus" to "humus" or "fur_frame_dogs" to "fur-frame" with a confidence score. This powers the auto-linking that saves hours of manual campaign-to-project mapping.

What I need:

SettingValueNotes
Server onetake-pg-west01 Azure Database for PostgreSQL Flexible Server
Parameter azure.extensions Server parameter (Networking → Server parameters)
Value to add pg_trgm Append to existing value (comma-separated if others exist)

Via Azure Portal:

  • Navigate to onetake-pg-west01 → Server parameters
  • Search for azure.extensions
  • Add PG_TRGM to the allowed list (check the checkbox)
  • Save

CLI equivalent:

az postgres flexible-server parameter set \
  --resource-group onetake-rg \
  --server-name onetake-pg-west01 \
  --name azure.extensions \
  --value pg_trgm
No restart required. Allowlisting an extension just enables CREATE EXTENSION — it doesn't install anything or require a server restart. I'll run CREATE EXTENSION IF NOT EXISTS pg_trgm after it's allowlisted.
